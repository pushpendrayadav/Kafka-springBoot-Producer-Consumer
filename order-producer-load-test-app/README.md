# Kafka Producer Load Test App

A Spring Boot application for load-testing Kafka producer **ACK modes** using [k6](https://k6.io/). It exposes multiple REST endpoints — each backed by a differently configured `KafkaTemplate` — so you can compare throughput, latency, and message durability side-by-side.

## Why This Exists

Kafka producers can be configured with different acknowledgement strategies. Choosing the right one is a trade-off between **speed** and **reliability**. This app lets you observe the differences under realistic load and simulated broker failures.

## ACK Modes Explained

| Mode                | `acks` | Retries | Idempotent | Speed   | Message Loss Risk                                    |
| ------------------- | ------ | ------- | ---------- | ------- | ---------------------------------------------------- |
| **Fire-and-Forget** | `0`    | `0`     | No         | Fastest | High — messages silently dropped                     |
| **ACK=0 + Retries** | `0`    | `3`     | No         | Fastest | High — retries are useless with `acks=0`             |
| **ACK=1**           | `1`    | `3`     | No         | Medium  | Low — lost only if leader crashes before replicating |
| **ACK=ALL**         | `all`  | `3`     | Yes        | Slowest | None\* — all replicas must confirm                   |

> \*As long as brokers recover within `delivery.timeout.ms` (30 s).

## Prerequisites

- **Java 17+**
- **Apache Kafka** running on `localhost:9092`
- **k6** installed (`brew install k6` on macOS)

## Getting Started

### 1. Start Kafka

Make sure Kafka (and ZooKeeper/KRaft) is running on `localhost:9092`.

### 2. Build & Run the App

```bash
cd order-producer-load-test-app
./mvnw spring-boot:run
```

The app starts on **port 8090** (so it can run alongside the main OrderService on 8080).

### 3. Run Load Tests

**Individual mode tests:**

```bash
k6 run k6-scripts/fire-and-forget-test.js
k6 run k6-scripts/ack0-test.js
k6 run k6-scripts/ack1-test.js
k6 run k6-scripts/ack-all-test.js
```

**Compare all modes in a single run:**

```bash
k6 run k6-scripts/compare-all-modes.js
```

This runs each mode for 30 s with 20 VUs, printing stats between scenarios.

## REST API

| Method | Endpoint                         | Description                                               |
| ------ | -------------------------------- | --------------------------------------------------------- |
| `POST` | `/api/load-test/fire-and-forget` | Send with `acks=0`, no callback                           |
| `POST` | `/api/load-test/ack-0`           | Send with `acks=0` + retries                              |
| `POST` | `/api/load-test/ack-1`           | Send with `acks=1` + retries                              |
| `POST` | `/api/load-test/ack-all`         | Send with `acks=all` + idempotent                         |
| `GET`  | `/api/load-test/stats`           | Get `totalSent` / `totalSuccess` / `totalFailed` counters |
| `POST` | `/api/load-test/reset`           | Reset counters for a new test run                         |

**Request body** (for POST endpoints):

```json
{
  "productName": "Laptop",
  "quantity": 2,
  "price": 999.99
}
```

`orderId` and `status` are auto-generated by the controller.

## Simulating Kafka Failures

To observe how each mode handles broker unavailability:

1. Start a load test (e.g., `compare-all-modes.js`).
2. While it's running, stop the Kafka broker:
   ```bash
   kafka-server-stop
   ```
3. Wait 5–10 seconds, then restart:
   ```bash
   kafka-server-start /opt/homebrew/etc/kafka/server.properties
   ```
4. After the test, check `/api/load-test/stats` to see which mode lost messages.

### Expected Results

| Mode            | Kafka Down Behaviour                               | On Recovery                                               |
| --------------- | -------------------------------------------------- | --------------------------------------------------------- |
| Fire-and-Forget | Messages silently lost, HTTP 200 keeps returning   | No retry                                                  |
| ACK=0 + Retries | Same as above — retries never trigger without acks | No retry                                                  |
| ACK=1           | Producer detects failure, retries kick in          | Messages delivered if broker recovers within retry window |
| ACK=ALL         | Same as ACK=1 but with idempotent deduplication    | No duplicates even after retries                          |

## k6 Test Profiles

Each individual test uses a **ramping VU** pattern:

| Phase      | Duration | VUs     |
| ---------- | -------- | ------- |
| Ramp up    | 10 s     | 0 → 10  |
| Steady     | 30 s     | 10      |
| Spike      | 10 s     | 10 → 50 |
| Hold spike | 20 s     | 50      |
| Ramp down  | 10 s     | 50 → 0  |

The `compare-all-modes.js` script uses **constant 20 VUs for 30 s** per mode, run sequentially.

## Configuration

All config is in `src/main/resources/application.yml`:

```yaml
server:
  port: 8090

spring:
  kafka:
    bootstrap-servers: localhost:9092
```

Producer properties (acks, retries, idempotence, timeouts) are set programmatically in `KafkaProducerConfig.java`.
